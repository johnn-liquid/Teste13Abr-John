## Definição

Skills são instruções específicas que agentes seguem para executar tarefas. São capacidades modulares que determinam **o que** o sistema pode fazer: análise de inadimplência, simulação de financiamento, avaliação de carteira, scoring de risco, etc. Skills consomem dados de conectores (D4), contexto do KB (D7) e entidades (D8) para produzir resultados.

---

## Princípios aplicáveis

**Agent First →** Skills são o cérebro especializado. O orquestrador (D6) é genérico — quem sabe analisar uma carteira é a skill. Sem skills, o produto é um chatbot genérico.

**Regulated Market →** Skills no contexto Liquid DEVEM garantir análises consistentes — respostas similares quando premissas iniciais são as mesmas. Diferente de um LLM genérico que pode variar a cada execução.

**Liquidity Driven →** Existem skills cujo propósito específico é avaliar se um ativo é transacionável, alimentando o pipeline de liquidez (D9).

---

## Regras de negócio

1. Existem dois tipos de skills: **Skills Liquid** (criadas e mantidas pela Liquid, operam em background, não configuráveis pelo usuário) e **Skills customizáveis** (criadas ou adaptadas pelo usuário para seu contexto).
2. Skills são invocadas pelo agente orquestrador (D6), não diretamente pelo usuário. O usuário expressa intenção; o orquestrador decide qual skill acionar.
3. Skills DEVEM declarar quais conectores necessitam. Se um conector não está ativado, o sistema oferece ativação inline (comportamento D4).
4. Skills DEVEM declarar quais inputs estruturados precisam do usuário (ex: CPF, renda, valor). O sistema renderiza esses inputs dinamicamente no chat — os campos vêm da skill, não da plataforma.
5. Na V1, todo Member com billing ativo acessa todas as skills disponíveis. Não há gate de permissão por skill.
6. Skills DEVEM retornar resultados em formato estruturado que o sistema pode renderizar como artefatos (dashboards, cards, gráficos).
7. O sistema DEVE suportar adição de novas skills sem deploy (mesmo requisito de D4 Connectors).

## Fixo vs. Variável

**FIXO:**

- Mecanismo de invocação: orquestrador identifica intenção → seleciona skill → skill declara requisitos → sistema coleta inputs → skill executa → sistema renderiza resultado
- Estrutura de declaração da skill: quais conectores precisa, quais inputs precisa, qual formato de saída
- Gestão de skills: listar, buscar, ativar/desativar (para customizáveis)

**VARIÁVEL:**

- Quais skills existem (catálogo cresce continuamente)
- Quais conectores cada skill requer
- Quais inputs estruturados cada skill solicita (campos, tipos, validações)
- Qual formato de resultado cada skill retorna
- Lógica interna de cada skill (a plataforma não sabe o que a skill faz por dentro)

**SLOTS:**

- Slot de skill: interface padronizada que toda skill implementa — declaração de requisitos (conectores, inputs), execução, formato de saída
- Slot de input estruturado: componente dinâmico no chat que renderiza campos definidos pela skill ativa

---

## Comportamentos esperados

- **Quando** o orquestrador identifica que uma skill é necessária, **o sistema** verifica requisitos (conectores ativos, billing, inputs necessários) antes de executar.
- **Quando** a skill precisa de inputs do usuário, **o sistema** renderiza inputs estruturados no chat com os campos, tipos e validações definidos pela skill.
- **Quando** a skill termina de executar, **o sistema** renderiza o resultado como artefato no formato declarado pela skill.
- **Quando** o usuário busca skills disponíveis, **o sistema** exibe o catálogo filtrado por localidade (D3) e contexto do Workspace.

---

## Casos de uso de referência

**Caso 1 — Skill com inputs dinâmicos** Usuário expressa intenção de análise. O orquestrador seleciona a skill adequada. A skill declara que precisa de 3 campos do usuário. O sistema renderiza esses campos como input estruturado no chat. O usuário preenche. A skill executa consumindo esses dados + dados de conectores. O resultado é um artefato renderizado na conversa.

**Caso 2 — Skill Liquid em background** Durante uma interação, o orquestrador aciona uma skill Liquid (interna) que o usuário não vê. Essa skill avalia o contexto acumulado e determina se há uma oportunidade de liquidez (D9). Se sim, o agente de liquidez é acionado. O usuário vê apenas o resultado — não a mecânica interna.

## Restrições

- NÃO são invocadas diretamente pelo usuário. Sempre mediadas pelo orquestrador.
- NÃO armazenam dados. Consomem e produzem — persistência é KB ou Entities.
- O design DEVE funcionar para 5, 50 ou 500 skills.
- V1 NÃO precisa de marketplace de skills entre empresas. Skills customizáveis são internas ao Workspace.
