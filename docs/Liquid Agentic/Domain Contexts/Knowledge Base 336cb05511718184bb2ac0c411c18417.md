## Definição

Knowledge Base é o repositório de contexto do usuário: arquivos, documentos, planilhas, dados extraídos de conectores. É o que personaliza a experiência, quanto mais contexto o usuário traz, mais relevantes são as respostas dos agentes. É também o mecanismo central de **fidelização**: quando o cliente traz dados sensíveis para dentro da plataforma, está investindo na relação.

---

## Princípios aplicáveis

**Liquidity Driven →** O KB é o ativo mais valioso da relação Liquid–cliente. Cada arquivo subido aumenta o switching cost e alimenta o pipeline de liquidez.

**Regulated Market →** O sistema DEVE rastrear quem subiu qual dado e quando. Dados sensíveis exigem tratamento conforme compliance.

---

## Regras de negócio

1. Na V1, o KB é uma **pasta global única por Workspace,** sem estrutura de projetos ou pastas complexa.
2. O usuário pode subir dados (arquivos, planilhas, documentos) a qualquer momento, via chat (como anexo) ou via área de Files (E4 Sidebar).
3. Agentes DEVEM conseguir consultar o KB durante a execução de skills. O KB é memória persistente do Workspace.
4. Dados de conectores (D4) PODEM enriquecer o KB automaticamente quando persistidos.
5. O KB é compartilhado no nível do Workspace. Todos os Members com acesso veem o mesmo KB.
6. O sistema DEVE distinguir "anexo" (arquivo subido pelo usuário) de "artefato" (output gerado pelo agente). São conceitos diferentes com ciclos de vida diferentes.

---

## Fixo vs. Variável

**FIXO:**

- Mecanismo de upload e processamento de arquivos
- Consulta do KB por agentes durante execução
- Distinção anexo vs. artefato
- Rastreabilidade de quem subiu o quê

**VARIÁVEL:**

- Quais tipos de arquivo o sistema processa (expandível: planilhas, PDFs, imagens, etc.)
- Quais dados cada Workspace acumula no KB
- Como o agente interpreta e estrutura os dados do KB (depende das skills)

---

## Comportamentos esperados

- **Quando** o usuário sobe um arquivo via chat ou via Files, **o sistema** processa e incorpora ao KB do Workspace.
- **Quando** um agente precisa de contexto para executar uma skill, **o sistema** consulta o KB relevante.
- **Quando** o usuário pergunta sobre "minha carteira" ou "meus dados", **o sistema** busca no KB do Workspace.

---

## Casos de uso de referência

**Caso 1 — Upload de contexto** Usuário sobe uma planilha com dados de negócio. O sistema processa e estrutura. Toda conversa futura sobre esses dados já tem contexto. O agente pode responder "com base nos dados que você subiu, o cenário X apresenta Y".

**Caso 2 — Enriquecimento via conector** O agente consulta um conector e retorna dados. O sistema persiste esses dados no KB, enriquecendo o contexto para futuras análises.

---

## Restrições

- V1 é pasta global única. Sem pastas, sem projetos dentro do KB.
- NÃO é um sistema de arquivos genérico. É contexto para agentes.
- NÃO processa dados em tempo real (streaming). Upload → processamento → disponível.
