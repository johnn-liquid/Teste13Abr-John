## Definição

Agents (Taxonomy) define os tipos de agentes que operam na plataforma, como se relacionam e como são apresentados ao usuário. Existem 4 tipos com funções distintas. Para o usuário, agentes são apresentados como **personas** ("Analista de Crédito") — internamente, são composições de skills e conectores orquestradas por um agente maestro.

---

## Princípios aplicáveis

**Agent First →** Princípio central. Toda a experiência passa por agentes. O produto não tem "telas de funcionalidades" — tem agentes que executam tarefas usando skills e conectores.

---

## Regras de negócio

1. Existem **4 tipos** de agentes com funções distintas (taxonomia abaixo).
2. O **Orchestrator** é o agente com quem o usuário interage. Ele interpreta intenção e aciona os demais.
3. Para o usuário, agentes são apresentados como **personas com nome e função** (ex: "Analista de Crédito"). Isso é mais interpretável do que expor a taxonomia técnica.
4. Na V1, o catálogo de agentes-persona DEVE ser enxuto e curado. O CEO disse: não adicionar muitos atalhos para evitar "erros prematuros".
5. Agentes NÃO são configuráveis pelo usuário na V1. São definidos pela Liquid.

---

## Taxonomia de Agentes

**Orchestrator (Maestro)**

- Agente principal de interação com o usuário
- Interpreta intenção, seleciona skills e conectores, coordena os demais agentes
- Sempre presente em toda conversa

**Skill Agents**

- Invocados por tarefa
- Executam skills específicas (D5)
- Consomem a skill, não a criam

**Entity Agents**

- Função específica, recorrente e persistente de mapear entidades do mercado (D8)
- Preenchem contexto interno da Liquid
- Formam dados proprietários
- Acumulam informações históricas
- Operam em background — o usuário não os aciona diretamente

**Liquidity Agents (Kernel Agents)**

- Função de identificar operações qualificáveis e fazer a ponte para processos de liquidez (D9)
- Avaliam se um ativo atende critérios de elegibilidade
- Quando identificam oportunidade, qualificam o cliente no pipeline de liquidez

---

## Fixo vs. Variável

**FIXO:**

- Taxonomia de 4 tipos com funções distintas
- Orchestrator sempre presente na conversa
- Mecanismo de apresentação como persona para o usuário

**VARIÁVEL:**

- Quais agentes-persona estão disponíveis no catálogo
- Quais skills e conectores cada agente-persona agrupa
- Quais entidades cada Entity Agent mapeia

---

## Comportamentos esperados

- **Quando** o usuário inicia uma conversa, **o sistema** conecta ao Orchestrator. Se o usuário escolheu um agente-persona (E4 Sidebar), o Orchestrator já tem contexto pré-carregado (skills e conectores relevantes).
- **Quando** o Orchestrator identifica necessidade de dados de entidade, **o sistema** aciona Entity Agents em background.
- **Quando** o Liquidity Agent identifica uma operação qualificável, **o sistema** muda o tom da conversa de informativo para operacional (D9).

---

## Casos de uso de referência

**Caso 1 — Agente-persona como atalho** Usuário clica numa persona na sidebar (ex: agente de análise). Nova task criada com contexto pré-carregado. O Orchestrator já sabe quais skills e conectores são relevantes. O usuário pergunta e recebe resposta sem precisar especificar ferramentas.

**Caso 2 — Entity Agent em background** Enquanto o usuário conversa sobre um tema de mercado, Entity Agents estão continuamente mapeando entidades relacionadas: atores, organizações, ativos. Esse contexto acumulado enriquece as respostas do Orchestrator e alimenta o Liquidity Agent.

## Restrições

- V1 NÃO permite criação de agentes customizados pelo usuário. Catálogo é definido pela Liquid.
- Entity Agents e Liquidity Agents NÃO são visíveis como personas para o usuário. Operam em background.
- O design NÃO deve expor a taxonomia técnica (Orchestrator, Skill Agent, etc.) ao usuário. Para o cliente, são personas.
