## Definição

Entities são abstrações estruturadas de objetos que se repetem no mercado e que o sistema precisa reconhecer e rastrear ao longo do tempo. Diferente do KB (dados brutos), entidades são **objetos com identidade persistente**: uma pessoa, uma organização, um ativo, um empreendimento. O sistema sabe que "Entidade X" é a mesma em diferentes conversas, skills e análises.

---

## Princípios aplicáveis

**Liquidity Driven →** Entidades são fundamentais para o score de liquidez (D9). Avaliar se um ativo é transacionável requer entidades estruturadas: quem é o ativo, quem é o comprador, qual o fluxo de pagamentos.

**Regulated Market →** Entidades carregam dados sensíveis (pessoas, operações financeiras). Rastreabilidade e integridade são obrigatórias.

---

## Regras de negócio

1. Entidades são criadas de duas formas: pelo **usuário** (direta ou indiretamente, ao subir dados que o sistema estrutura) ou por **Entity Agents** (D6) que mapeiam entidades do mercado continuamente em background.
2. Entidades criadas por Entity Agents formam **dados proprietários da Liquid,** são inteligência acumulada do sistema.
3. Uma entidade PODE estar envolvida em múltiplas conversas/análises simultaneamente. O sistema DEVE manter a consistência dos dados.
4. Entidades DEVEM permitir correlações entre si (ex: uma pessoa está vinculada a uma organização que possui um ativo).
5. Entidades são vinculadas ao Workspace. Visíveis para todos os Members.

---

## Fixo vs. Variável

**FIXO:**

- Mecanismo de criação, atualização e consulta de entidades
- Persistência com identidade única (a entidade "existe" independente de conversas)
- Correlação entre entidades
- Rastreabilidade (quem criou, quando, com quais dados)

**VARIÁVEL:**

- Quais tipos de entidade existem (expandível: pessoas, organizações, ativos, empreendimentos, carteiras, etc.)
- Quais campos cada tipo de entidade tem
- Quais Entity Agents estão ativos e o que mapeiam

**SLOTS:**

- Slot de tipo de entidade: o sistema aceita novos tipos de entidade conforme o mercado demanda, sem redesenho

---

## Comportamentos esperados

- **Quando** o usuário sobe dados que referenciam um objeto identificável, **o sistema** cria ou atualiza a entidade correspondente.
- **Quando** um Entity Agent (D6) mapeia informações do mercado, **o sistema** cria/atualiza entidades em background.
- **Quando** uma skill (D5) precisa de dados de uma entidade, **o sistema** fornece o contexto acumulado.
- **Quando** a mesma entidade aparece em conversas/skills diferentes, **o sistema** reconhece e mantém o contexto unificado.

---

## Casos de uso de referência

**Caso 1 — Entidade criada a partir de dados do usuário** Usuário sobe dados com referência a um ator específico. O sistema cria a entidade, correlaciona com dados de conectores e KB. Em futuras conversas, o agente já conhece essa entidade.

**Caso 2 — Entity Agent mapeando mercado** Entity Agents mapeiam continuamente atores e ativos de um segmento. Quando o usuário pergunta sobre o mercado, as entidades já existem com dados acumulados, a resposta é rica e atualizada.

## Restrições

- NÃO são visíveis como "tela de entidades" para o usuário na V1. O usuário interage com entidades indiretamente via conversa com agentes.
- NÃO substituem o KB. KB é contexto bruto; entidades são objetos estruturados com identidade.
- Conflitos de dados (dois corretores sobem informações diferentes sobre a mesma entidade) precisam de resolução [EM ABERTO].
