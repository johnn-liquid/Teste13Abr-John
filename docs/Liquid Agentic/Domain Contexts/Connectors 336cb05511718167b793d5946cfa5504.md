## Definição

Connectors são integrações com fontes de dados externas (abertas e fechadas) que permitem aos agentes acessar, consultar e processar informação do mundo real. São os provedores de dados que alimentam skills e agentes. O usuário ativa e gerencia conectores; o agente os invoca durante a execução de skills.

---

## Princípios aplicáveis

**Regulated Market →** Toda consulta via conector DEVE ser rastreável (quem, quando, qual dado, qual resultado). Conectores acessam dados financeiros sensíveis.

**Global →** Conectores são dependentes de localidade (D3). O sistema DEVE exibir apenas os conectores relevantes para a localidade do Workspace.

**Liquidity Driven →** Conectores são a matéria-prima do score de liquidez (D9). Sem conectores, não há análise, não há transação, não há receita.

**PLG (Product-Led Growth)** Ativação de conectores DEVE ser self-service quando possível.

**Agent First →** Conectores são consumidos por agentes, não diretamente pelo usuário. O usuário ativa; o agente invoca.

---

## Regras de negócio

1. O usuário NUNCA interage diretamente com a API de um conector. Toda interação é mediada por agente/skill.
2. O sistema DEVE suportar a adição de novos conectores sem necessidade de deploy.
3. Conectores com custo por consulta DEVEM estar vinculados ao billing (D2). Sistema DEVE impedir consultas se crédito insuficiente.
4. Todo conector DEVE ter status visível: disponível, ativo, em uso, com erro.
5. Conectores NÃO possuem segmentação por categoria — a organização é por tags.
6. Conectores DEVEM ser compartilháveis entre múltiplos usuários do mesmo Workspace. Credencial pode ser do Workspace, não do indivíduo.
7. Conectores DEVEM respeitar a localidade do Workspace (D3).
8. Quando uma skill requer um conector não ativado, o sistema DEVE oferecer ativação inline (dentro do chat), sem redirecionar para Settings.

---

## Fixo vs. Variável

**FIXO:**

- Sistema de gestão: listar, ativar, desativar, exibir status, filtrar por tag e localidade
- Ciclo de vida: disponível → ativo → em uso → com erro
- Mecanismo de ativação: solicitar credenciais/autorização, validar conexão
- Feedback de erro quando conector falha
- Rastreabilidade de cada consulta

**VARIÁVEL:**

- Quais conectores existem no catálogo (cresce continuamente)
- Quais campos de credencial/autorização cada conector exige (OAuth, API key, upload, etc.)
- Quais dados cada conector retorna (schema variável)
- Quais tags cada conector tem
- Em quais localidades cada conector está disponível
- Custo por consulta de cada conector

**SLOTS:**

- Slot de conector: interface padronizada que qualquer novo conector implementa — entrada (credenciais), saída (dados estruturados), metadados (nome, descrição, tags, localidade, custo)
- Slot de ativação inline: quando o agente precisa de um conector não ativado, o sistema renderiza o fluxo de ativação dentro da conversa

## Comportamentos esperados

- **Quando** o usuário acessa a área de Conectores, **o sistema** exibe a lista filtrada por localidade, com status de cada um.
- **Quando** o usuário ativa um conector, **o sistema** solicita credenciais/autorizações necessárias e valida a conexão.
- **Quando** um agente precisa de conector não ativado, **o sistema** oferece ativação inline no chat.
- **Quando** um conector falha durante execução, **o sistema** notifica o usuário com causa e ação sugerida.
- **Quando** a Liquid publica um novo conector (D10), **o sistema** o disponibiliza automaticamente para Workspaces com localidade compatível.

---

## Casos de uso de referência

**Caso 1 — Ativação self-service** Usuário aciona uma skill que requer um conector ainda não ativado. O sistema informa qual conector é necessário e oferece ativação inline. O usuário autoriza, o sistema valida, a skill continua executando.

**Caso 2 — Múltiplos conectores numa análise** Uma skill de análise combina dados de três conectores: um de dados públicos (gratuito), um de bureau (pago) e um de dados do próprio usuário (upload). O sistema verifica billing para o conector pago, executa os três, e entrega o resultado integrado.

**Caso 3 — Novo mercado** A Liquid expande para um novo país. Novos conectores são criados com a tag de localidade correspondente (D10 Platform Operations). Workspaces com essa localidade passam a vê-los automaticamente.

## Restrições

- NÃO executam lógica de análise. Apenas acessam e retornam dados. Quem processa é a skill.
- NÃO armazenam dados permanentemente. Dados retornados vão para KB ou Entities se persistidos.
- NÃO são API pública. Internos ao sistema.
- NÃO substituem uploads manuais. Upload é sempre alternativa disponível.
- O design DEVE funcionar para 5, 50 ou 500 conectores. Componentes de listagem e busca DEVEM escalar.

## Glossário local

| Termo | Definição |
| --- | --- |
| **Conector / Connector** | Integração com fonte de dados externa. Não executa lógica — acessa e retorna dados. |
| **Fonte aberta** | Dado público acessível sem autenticação especial. |
| **Fonte fechada** | Dado que requer credencial, contrato ou pagamento. |
| **Tag** | Rótulo descritivo para organização. Não é categoria hierárquica. |
| **Ativação** | Processo de configurar e habilitar um conector no Workspace. |
| **Ativação inline** | Fluxo de ativação de conector oferecido dentro do chat quando o agente precisa. |
